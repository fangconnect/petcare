# Project Context: Pet Care Super App (Chronic Disease Management)

## 1. Project Overview
We are building a High-Performance MVP app for tracking pet chronic diseases (e.g., CKD, Diabetes).
The core feature is a **Dynamic Logging System** where the logging UI changes based on the pet's specific disease configuration.

## 2. Tech Stack
- **Backend:** Golang (Fiber Framework), GORM (ORM), PostgreSQL.
- **Frontend:** Flutter (Dart), Riverpod (State Management), Dio.
- **Architecture:** Clean Architecture (Domain -> Usecase -> Repository -> Delivery).

## 3. Database Schema (PostgreSQL) - CRITICAL
We use a "Master Data" approach to allow dynamic form generation.

**Tables:**
1.  `users`: id (UUID), email, password_hash, created_at.
2.  `pets`: id (UUID), user_id (FK), disease_id (FK), name, image_url, birthdate.
3.  `diseases` (Master): id (int), name (e.g., "Kidney Disease"), description.
4.  `activities` (Master): id (int), name (e.g., "Saline IV", "Dry Food"), unit (e.g., "ml", "g"), input_type (enum: "number", "text", "checkbox").
5.  `disease_activities` (Config): disease_id (FK), activity_id (FK), is_required (bool).
6.  `logs` (Transaction): id (UUID), pet_id (FK), activity_id (FK), value (decimal), note (text), recorded_at (datetime).

## 4. Key Business Logic
- **Admin** configures `diseases` and `activities`.
- **User** selects a `disease` for their pet.
- **App** fetches `disease_activities` to render the correct input buttons for that pet.
